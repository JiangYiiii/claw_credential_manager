# 本机凭证与 Token 管理服务 — 需求与架构说明

**文档版本**：0.2  
**日期**：2026-04-22  
**状态**：产品决策已锁定，可进入实现

---

## 1. 背景与范围

### 1.1 目标

从零实现一个**纯本机**的账号密码与登录 Token 管理服务，集中保存网站/服务的账号密码、API Key，以及登录后获得的 Token（如 access token）。主要消费者为本机运行的 OpenClaw 或其他本地 Agent，通过 **HTTP**（仅 `127.0.0.1`）与/或 **本地 MCP** 访问。

**第一版不包含**：跨设备同步、浏览器扩展、自动填表、将服务暴露到局域网/公网、替代企业级 KMS/HSM。

### 1.2 Token 续期策略（第一版）

完整通用 OAuth 等协议不在第一版实现。约定为**可配置的刷新脚本**：首次可由人工旁路或 Agent 生成固定脚本（如 Playwright/curl），之后由本机调度重复执行脚本，将新 Token 写回对应 Entry 字段。

## 2. 架构原则（约束）

| 编号 | 原则 |
|------|------|
| P1 | 所有监听地址必须为环回地址（默认 `127.0.0.1`），**默认禁止** `0.0.0.0` / 全网卡监听。 |
| P2 | MCP 与 HTTP 为可选入口；两者均支持**最小权限**（例如按 `entry_id` / 别名 allowlist），**禁止**无策略的「全库导出」。 |
| P3 | 密钥材料默认不落日志、不进入模型上下文；除非调用方显式请求且经策略允许（策略可后续迭代）。 |
| P4 | 存储层抽象为 `VaultBackend`：**第一版实现 KeePass `.kdbx`**；另预留扩展点（如自研加密存储），文档说明迁移与第二后端接入方式。 |
| P5 | 主密码/解锁：**已选 B2**——允许从**受限文件或环境变量**注入复合密钥材料（或等效会话解锁）；README 必须写明无人值守收益与密钥落盘/泄漏风险；交互式主密码可作为补充能力，非默认路径。 |

## 3. 技术选型要点

### 3.1 实现语言：Go（推荐）或 Rust

| 维度 | Go | Rust |
|------|-----|------|
| 可维护性 | 生态成熟，HTTP/MCP 服务端示例多，迭代快 | 内存安全强，适合长期安全敏感组件 |
| 加密与 SQLite | `crypto/*`、`modernc.org/sqlite` 等开箱即用 | `ring`/`rusqlite` 等组合成熟 |
| MCP | 可参考现有 stdio MCP 服务模式 | 同样有成熟模式 |
| 团队偏好 | 若团队 Go 居多，优先 Go | 若强调极致安全边界可选 Rust |

**建议**：若无强约束，**第一版选用 Go**：交付速度与本地 Agent 集成文档生态更友好；Vault 边界清晰后 Rust 重写可作为远期选项。

### 3.2 传输与 TLS

- **默认**：HTTP + 仅 localhost，多数场景足够（攻击面为本机进程）。
- **可选**：本机 TLS（自签证书）；需在 README 中写明生成证书、信任根、客户端校验方式。
- **原则**：不因「用了 TLS」而放宽绑定地址——仍仅限环回。

### 3.3 存储后端（已锁定）

| 方案 | 优点 | 缺点 |
|------|------|------|
| **自研加密 SQLite/文件**（扩展，非 MVP） | schema 与 Entry 模型完全可控 | 需自行严谨实现 KDF、加密套件、格式版本化 |
| **KeePass `.kdbx`**（**MVP**） | 加密格式经长期检验，可与 KeePassXC 等共用文件 | 字段映射、自动化写回需约定；需锁定读写库版本 |

**第一版**：实现 **KeePass `.kdbx` 的 `VaultBackend`**；自研加密存储仅在接口层保留扩展说明，不作为首版交付。

### 3.4 密钥派生与加密（方向性）

- `.kdbx` 的开锁：由 KeePass 格式定义的 **composite key**（密码、密钥文件、可选 Windows 用户账户等）；**B2** 指通过**环境变量或受限文件**向进程提供上述材料（等价于无人值守解锁），而非在应用内重做一套 KDF 替代 `.kdbx` 内置流程。
- 条目内密码/字段由 KDBX 存储层保护；若在自定义字段存放 token 与脚本路径，仍属 KDBX 加密载体之内。
- 选用的 Go/Rust KeePass 库版本与支持的 KDBX 版本（如 3.1 / 4）在实现阶段锁定并写入 README。

### 3.5 MCP 传输

- **stdio MCP**：典型本地 Agent 挂载方式，无网络监听，符合「不对外发布」。
- 若未来支持 **SSE/HTTP MCP**：仍必须仅限 `127.0.0.1`，并与 HTTP API 共用鉴权与 allowlist。

## 4. 模块划分

```
┌─────────────────────────────────────────────────────────┐
│              Transport Layer（可选组合）                   │
│  • HTTP Server（127.0.0.1）  • MCP（stdio / 可选本地 HTTP） │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  API / MCP Handler                                         │
│  • 路由、请求校验、速率限制、失败锁定                       │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  AuthZ / Policy                                           │
│  • API Key 或 mTLS（可选）• Entry allowlist • 审计钩子      │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  Credential Service（领域逻辑）                           │
│  • CRUD Entry • 解密敏感字段仅在授权路径 • 刷新脚本调度    │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  VaultBackend（接口）                                      │
│  • KDBX 适配器（MVP）• 自研加密存储（扩展预留）               │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  Audit（可选简化）                                         │
│  • 记录：主体、时间、entry_id、操作类型；不落明文           │
└─────────────────────────────────────────────────────────┘
```

| 模块 | 职责 |
|------|------|
| **Transport** | HTTP 仅绑定环回；MCP 进程生命周期与 Agent 一致；统一接入日志脱敏。 |
| **AuthZ / Policy** | **C1**：单一服务级 API Key + **全局** entry allowlist；禁止无 allowlist 的「读全库密文」。 |
| **Credential Service** | 列表（可不含密文）、按 id 读取（含解密）、写入/更新、触发刷新脚本。 |
| **VaultBackend** | 持久化、加密、版本迁移钩子；第二后端通过新实现注入。 |
| **Security Controls** | 速率限制、失败锁定、配置与数据文件权限文档化。 |
| **Scheduler / Runner（可选 MVP）** | 定时或手动触发 `refresh_script_path`，stdout/约定格式解析后写回 token。 |

## 5. 数据模型（建议）

### 5.1 Entry

| 字段 | 说明 |
|------|------|
| `id` | 稳定标识（UUID 或后端生成） |
| `name` | 人类可读名称 |
| `type` | `password` / `token` / `mixed` |
| `username` | 可选 |
| `password` | 敏感；列表接口默认不返回 |
| `custom_fields` | JSON；可含 API Key、额外 header 等；敏感键标记由实现约定 |
| `tags` | 标签列表 |
| `notes` | 备注（是否加密由 Vault 策略决定） |
| `metadata` | 如 `token_expires_at`、`refresh_script_path`、`last_refreshed_at` 等 |

### 5.2 审计（第一版可简化）

- 记录：时间、`entry_id`、操作（list / get / put / refresh）；C1 下可为「同一服务 key」或固定 `client=service`。
- **不记录**明文密码或 token。

## 6. API / MCP 能力（第一版最小集）

| 能力 | HTTP | MCP（概念对齐） | 说明 |
|------|------|-----------------|------|
| 列出条目 | ✓ | ✓ | 返回元数据，**默认不含**密文字段 |
| 按 id 获取单条 | ✓ | ✓ | 含解密后的敏感字段；**必须**通过权限校验 |
| 写入/更新条目 | ✓ | ✓ | 含脚本路径与过期时间等 |
| 触发刷新脚本 | 可选 MVP | 可选 MVP | 执行绑定在 entry 上的脚本并写回 token |

## 7. 安全与加固清单

- **监听**：默认 `127.0.0.1`；配置若允许其他地址须在文档中标为「不推荐」并需显式开关。
- **速率限制与失败锁定**：防止本机恶意进程对 API Key 或解锁流程爆破（阈值可配置）。
- **日志**：禁止记录解密后的密钥、完整 Authorization 头；调试模式单独开关。
- **文件权限**：Unix 下数据库文件、配置目录建议 `0600`/`0700`；README 写明 `umask` 与容器内用户。
- **TLS**：可选；自签证书的信任与轮换在 README 说明。

## 9. 第一版里程碑（MVP）

### M1：核心存储与解锁

- 实现 **`VaultBackend`：KeePass `.kdbx`**（读/写与格式版本见 README）。
- **解锁**：支持从**受限密钥文件路径**与/或**环境变量**注入 composite key 材料（B2）；可选补充交互式密码启动，文档说明风险与文件权限（`0600`/目录 `0700`）。
- 本地 `.kdbx` 路径与权限说明；**不得**默认监听非环回地址。

### M2：HTTP API（仅 127.0.0.1）

- 列出（无密文）、按 id 获取（含密文）、更新条目。
- **C1**：单一服务级 API Key + **全局** entry allowlist。
- 基础速率限制与失败锁定。

### M3：MCP（stdio）

- 与 HTTP 对齐的最小工具集；同样受 allowlist 约束。

### M4：文档与可运维性

- README：威胁模型、初始化、OpenClaw 集成、备份恢复、TLS 可选说明。

### M5（可选，仍属第一版范围若时间允许）

- 触发刷新脚本并写回 token；简单调度（单次触发 + 可选 cron 式配置）。

**明确不纳入 MVP**：跨设备同步、浏览器插件、通用 OAuth 刷新、全功能审计面板。

## 10. 威胁模型（摘要，供 README 展开）

| 假设 | 说明 |
|------|------|
| 信任边界 | 本机用户会话内进程可互相尝试连接 `127.0.0.1`；服务需缓解未授权读取与爆破。 |
| 非目标 | 不防御物理机被攻陷后的内存dump；不防御 root 恶意软件直接读进程内存。 |
| 设计指向 | 最小权限 API、加密静态存储、日志与模型侧默认不泄漏密文。 |

## 11. 已确认的产品 / 安全决策

以下结论已锁定，实现与配置文件格式应与之对齐。

| 决策项 | 选项 | 备注 |
|--------|------|------|
| **A · Vault 后端** | **A2**：KeePass `.kdbx` 作为主存储 | `VaultBackend` 首版仅实现 KDBX；自研加密存储为后续扩展。 |
| **B · 解锁方式** | **B2**：允许从**受限文件**或**环境变量**注入会话/复合密钥材料 | README 必须强调密钥落盘与进程环境可见性风险；推荐文件权限与「不入备份/不入同步目录」清单。可选保留交互式密码作为辅助入口。 |
| **C · 鉴权模型** | **C1**：**单一服务级 API Key** + **全局 entry allowlist** | HTTP 与 MCP 共用同一策略；审计字段可不区分多客户端（或记固定标签）。 |

**实现侧影响摘要**

- **A2**：需在文档中固定 **Entry ↔ KDBX 条目/自定义字段映射**（如 `refresh_script_path`、`token_expires_at` 存放位置）。
- **B2**：配置项需包含密钥文件路径、环境变量名、以及与 `.kdbx` 开锁方式（密码-only、keyfile-only、组合）的对应关系。
- **C1**：仅需维护一个 API Key 机密与一份 allowlist；禁止默认「allowlist 为空即允许全部」（若采用空列表语义，须在评审中明确为「拒绝全部」或「开发模式专用」并打标）。
